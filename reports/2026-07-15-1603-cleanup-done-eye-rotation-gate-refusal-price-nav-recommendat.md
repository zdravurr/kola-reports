## 1. ДЕШЁВАЯ ЧИСТКА — СДЕЛАНО (доказано)
### 1a. Ротация eye_* скринов (диск не ползёт)
- Корень: `eyes_capture(tag)` в inbox_watcher писал `eye_{tag}_{ts}.png` и не чистил; `cleanup_temp`
  (по возрасту 1ч) в этом процессе не звался → eye_sms дорос до **95 шт / 52МБ**.
- Фикс: **cap-on-write** — `_prune_eyes(tag, keep=15)` вызывается после каждого снимка. Держит последние
  15 на тег независимо от того, кто и как часто снимает. Потолок ~7-8МБ на тег.
- Доказано РЕАЛЬНОЙ функцией: бэклог eye_sms **95→15 шт, 52МБ→8.6МБ**; дальше кап держится сам.

### 1b. Гейт не режет цену-ОТКАЗ («$60 too low»)
- Корень: `contradictions()` ловил любую не-$100 сумму; исключение смотрело только ДО числа, а маркер
  отказа («too low») стоит ПОСЛЕ → «$60 is too low» ложно блокировался.
- Фикс: окно И ДО, И ПОСЛЕ суммы + маркеры отказа/встречного (too low/high/much, can't do, won't go,
  no lower, below my/the, you offer/counter).
- Доказано 5 кейсами: «$60 too low» / «can't do $60» / «you offered $60» → **clean**; «let it go for $60» /
  «it is $60» (реальная неверная цена) → **по-прежнему FLAGGED**. Точно, дыру в фактчеке не открыли.

## 2. NAV-ФЛАП / adb-ssh — ЧЕСТНАЯ ИНЖЕНЕРНАЯ РЕКОМЕНДАЦИЯ
**Что это:** список тредов Messenger иногда не отрисовывается (kind=unknown), ssh-мастер/adb таймаутят
(60с), «заход упал/таймаут rc=0» в конце прохода, Michelle зависает → breaker MAX_FAILS=2 → человеку.
**Корень:** мы рулим ЖИВЫМ потребительским телефоном по Tailscale (Doze, прыжки relay↔direct, Termux sshd)
+ флак Android-UI (рендер Messenger). Это НЕ баг в нашем коде — это природа удалённого управления UI телефона.
**Что уже держит:** recover_phone (сносит протухший мастер, реконнект adb, будит экран), breaker (уводит к
человеку, НИКОГДА не действует вслепую), 480с-таймаут прохода, тёплый ssh-мукс с ретраем. **Денег НЕ теряем** —
сбойный read → эскалация человеку, покупатель не пропадает.

**МОЯ РЕКОМЕНДАЦИЯ: оставить предохранители держать и ИДТИ В ТРЕЙДИНГ-ЛИД.** Полный «правильный фикс»
(сделать удалённый UI телефона 100% надёжным) — кроличья нора с убывающей отдачей. Стоимость флапа сейчас =
несколько лидов/день к человеку + шум в логах, НЕ потерянные деньги.
**НО** есть 2-3 ДЕШЁВЫХ точечных улучшения на потом (bounded, ~1 сессия, НЕ большой проект):
(a) проактивный recover_phone в НАЧАЛЕ цикла, если прошлый упал — греем связь до захода;
(b) render-retry на kind=unknown (подождать 2-3с + пере-dump до вердикта «не отрисовалось»);
(c) диагностировать и заглушить rc=0 в конце прохода (понять ненулевой выход — убрать шум).
Это НЕ блокер трейдинг-лида. Приоритет — твой; мой инженерный лин: трейдинг-лид вперёд, эту тройку — как
мелкую уборку позже.

## 3. ПЛАН: ТРЕЙДИНГ-ЛИД (надзиратель Titan + Solana) — ТОЛЬКО ПЛАН, НИЧЕГО НЕ СТРОИЛ
Модель — как Дирижёр: отдельный изолированный контейнер, свой Opus 4.8 CLI, свой Telegram, БЕЗ root.
Но роль иная: **чистый НАБЛЮДАТЕЛЬ, только чтение. Никогда не торгует, не трогает ключи, не стартует/стопит боты.**

### 3.1 Контейнер и изоляция (стены — как у Дирижёра, даже строже)
- Отдельный контейнер `trader-lead`, пользователь uid 1000 (НЕ root), CapEff=0.
- НЕТ: sudo, su-root, systemctl, docker.sock, docker-cli → **физически не может стартовать/стопить боты**.
- НЕТ доступа к ключам: `.env` бирж (BingX/OKX/Bybit), Anthropic-ключ трейдинга — НЕ монтируются.
- Свой Opus 4.8 CLI на томе (.tools, свои creds 600, gitignored) — как в dirigent, персистентно, переживает рестарт.
- Свой Telegram-бот/канал (отдельный токен) — только для алармов Боссу.

### 3.2 БЕЗОПАСНАЯ ГРАНИЦА ЧТЕНИЯ (ответ на твой ключевой вопрос)
Как читать статус БЕЗ записи и БЕЗ ключей:
- Боты и так пишут состояние на диск: `trades.db` (SQLite: сделки, позиции, PnL), лог-файлы, health-json.
- Лиду даём **ТОЛЬКО read-only bind-mount** ровно этих артефактов: titan `trades.db`, sol `trades.db`,
  их логи, health-json. Больше НИЧЕГО из /root. **`.env`/ключи НЕ монтируются вовсе.**
- SQLite читаем БЕЗОПАСНО для ЖИВОЙ БД: самый чистый способ — лид периодически делает **read-only СНИМОК**
  (копию) БД и читает копию, НИКОГДА не касаясь живого файла/локов бота. (Альтернатива: SQLite `mode=ro`
  с аккуратной WAL-обработкой — валидируем на этапе сборки; у read-only WAL есть нюанс с -shm, поэтому
  снимок надёжнее.) Если снимок живой WAL-БД окажется «рваным» — МИНИМАЛЬНАЯ аддитивная правка ботов =
  крошечный статус-экспорт JSON, но это ОТЛОЖЕНО (сейчас Titan/SOL не трогаем).
- Из этих данных лид считает: открытые позиции, PnL (реализ./нереализ.), просадку, время последней сделки,
  живость сервиса (по свежести health-json / записей БД — read-only), всплески ошибок из логов.

### 3.3 Что делает и чего НЕ делает
- **Делает:** свой Telegram-аларм на: бот упал (health-json протух / БД не пишется), просадка сверх порога,
  плохой ран (серия убытков), нет хартбита, всплеск ошибок. Opus CLI помогает СФОРМУЛИРОВАТЬ/оценить
  «это реальная проблема?» (нюанс), но это ТОЛЬКО текст Боссу.
- **НЕ делает (физически не может):** торговать (нет ключей, нет аутентификации на бирже), стартовать/стопить
  боты (нет root/systemctl/sock), писать в trades.db (ro-mount), трогать `.env`. **Рычаг остаётся у Босса —
  ноль авто-вмешательства.**

### 3.4 Сеть
Egress только для Telegram + Anthropic (Opus). Доступ к API бирж НЕ нужен и не даётся (а без ключей и
бесполезен). По желанию — ограничить egress файрволом до этих двух хостов (defense-in-depth).

### 3.5 АККАУНТ (твой вопрос)
Рекомендую **ОТДЕЛЬНЫЙ Claude-аккаунт для трейдинг-мира** — полная изоляция от sales-Max-аккаунта
(сейчас host+агенты+dirigent на одном). Тогда компрометация/лимиты одного мира не задевают другой.
Если решишь отложить — временно стартуем с копией токена (как dirigent), но это шаг назад от цели «не делиться».
Это твоё решение; в плане отмечено.

### 3.6 Порядок сборки (когда одобришь — по шагам, каждый доказан живьём)
1) Контейнер + изоляция (доказать стены: нет root/sock/systemctl/keys, CapEff=0).
2) Read-only снимок trades.db обоих ботов + парс статуса (доказать: читаю позиции/PnL, НЕ могу писать).

4) Пороги алармов (просадка/бот-даун/серия убытков) на РЕАЛЬНЫХ данных, dry-run сначала.
5) Прогон живьём: подсунуть «бот-даун»/«просадка» → аларм пришёл, вмешательства НЕТ.
Titan/Solana всё это время — только чтение, ноль записи.
