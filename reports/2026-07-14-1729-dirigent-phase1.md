### 3. ONE OWNER OF THE PHONE — the risk you named, closed at the transport

Before today there was **no screen lock at all**. Host crons, agent hands and my manual sessions each reached the Pixel on their
own. Two hands on one screen do not produce an error — they produce **silence**: "the thread list did not render", a tap lands on
the wrong element, a reply lands in the wrong thread. That is how last session died.

`lib/phone_lock.py` — a `flock` on `.kola_state/locks/pixel.lock`. That file lives on the volume that is bind-mounted into the
container, so host and container hold **the same inode** → mutual exclusion **across the container boundary**, not merely between
host processes. I used `flock` and not a PID file because the kernel releases it even on SIGKILL/OOM — a hand-rolled lock would
have wedged the phone forever after the next OOM kill. **Fail-closed:** cannot take the lock → the operation does **not** run.

It is wired into `pixel_ssh.run / screencap / ui_dump` — the one chokepoint (`phone_hands`, `kola_hands` and the sweeps all
delegate there). Discipline cannot hold this. A lock can.

Proven live, in both directions:

```
host takes a real screenshot through the lock       -> OK (1.9 MB PNG)
second driver tries while the first holds           -> BLOCKED (fail-closed)
CONTAINER holds the phone, HOST tries to drive it   -> HOST BLOCKED by the container's lock
container releases                                  -> host works again immediately
```

**Known gap, stated plainly:** the X/FB warmup scripts (`social_x`, `x_tab_guard`, `kola_eyes`) still reach `adb` directly and do
not yet take the lock. They are all `#PAUSED` today, so nothing is racing right now — but **they must take the lock before the X
warmup is ever un-paused.** I did not touch them in this phase.
