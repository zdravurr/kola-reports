# ШАГ 1 — Pixel + X CDP мост: ДИАГНОСТИКА (16.07.2026, 12:2x UTC)

## ГЛАВНОЕ В ОДНОЙ СТРОКЕ

**Исходная гипотеза НЕ ПОДТВЕРДИЛАСЬ. Pixel ЗДОРОВ. Деградации нет. Ребут НЕ делал — данные его прямо опровергают.**

Мост умирает не от «ночной деградации», а потому что **Chrome уходит в фон**: продажный
свип (`fb_inbox_sweep`) владеет экраном и выталкивает Chrome → Android перестаёт обслуживать
`@chrome_devtools_remote` → CDP мёртв. Chrome на переднем плане — CDP отвечает мгновенно.
**Доказано причинно, а не корреляцией** (см. «Опыт» ниже).

А X-прогрев (70ч) стоит вообще по ДРУГОЙ причине и **работает как задумано**:
**36 покупателей ждут** → флаг `buyer_waiting` взведён → прогрев сам уступает Сене/Боре
(«деньги > прогрев»). Это не поломка. Это приоритет, который Босс сам и приказал.

## КОРНИ (три независимых, ни один — не «деградация Pixel»)

### 1. Два крона дерутся за экран, и «healed» — ЛОЖЬ

`pixel_keepalive.sh` (`*/15`) и `fb_inbox_sweep.py` (`*/15`, `timeout 480`) стоят на
**одних и тех же минутах**. Свип выигрывает (живёт минутами, гоняет FB), Chrome улетает
в фон, CDP умирает. Следующий тик keepalive пишет «bridge healed» — и так по кругу.

Но вот что важнее. Вот сам код:

```bash
if ! curl -s -m4 http://localhost:19222/json/version >/dev/null 2>&1; then
    adb connect ... ; adb forward ... ; ssh 'am start ... chrome'
    echo "[keepalive] $(date -u +%H:%M) bridge healed (adb+forward+chrome)"
fi
```

Он печатает **«healed» СРАЗУ после попытки починки, НЕ перепроверив CDP**.
То есть «healed» на самом деле значит: **«CDP лежал, я подёргал ручки, вышло или нет — не знаю»**.

Отсюда и «зажило 40% тиков (39/96)» из утреннего отчёта — цифра означает не «зажило»,
а **«лежало»**. И сейчас уже не 40%: 11:00, 11:15, 11:30, 11:45, 12:00 — **healed КАЖДЫЙ тик**,
то есть CDP лежит непрерывно больше часа. Это тот же класс бага, что и `RC=0` у EV-группы:
**зелёный рапорт без проверки факта.**

### 2. X-прогрев (70ч): уступает продажам — ПО ЗАДУМКЕ, и вкладки X больше НЕТ

В логе `zdravurr_cron.log` мост поднимался **успешно**, и прогрев всё равно не стартовал:
```
[bridge] up (try 1)
  [x-warmup] 💰 клиент ждёт на телефоне — НЕ стартую прогрев, уступаю Сене/Боре
[bridge] up (try 4)     → тот же уступ
[bridge] already up     → тот же уступ  (×4 захода)
```
Причина — `client_pending.json` = `{"buyer_waiting": ...}`, а свип прямо сейчас видит
**`тредов=112 ЖДУТ=36`**. **36 живых покупателей ждут ответа.** Флаг взведён законно.
Последняя настоящая сессия — «Session start 10:00 UTC» (13.07), дальше каждый заход уступал.

**Это не баг. Это «деньги > прогрев», как приказано.** Чинить тут нечего — вопрос к Боссу
политический: 36 ждущих покупателей будут душить X-прогрев столько, сколько будут ждать.

**Но есть и техническая дыра:** в живом CDP **НЕТ НИ ОДНОЙ вкладки x.com**. Совсем.
11 вкладок — craigslist, google account, nextdoor, tcgsherpa, две FB-логин — и **ни одной X**.
Значит даже если снять уступ, `zdravurr_auto` не к чему подключаться: composer-вкладка,
на которой всё держалось, **исчезла**. Это ровно то, о чём предупреждал
`bug_zdravurr_ca_cdp_tab32_timeout` («tab ids change — never hardcode»), только теперь
пропал и fin-таб. Восстановление вкладок — шаг 2, вслепую не трогаю (риск кросс-контаминации
cookie между fin и ca = потерять оба прогреваемых аккаунта).

### 3. FB EV-группа: я едва не соврал — проверил и НЕ подтверждаю «разлогин»

Увидел в Chrome две вкладки `m.facebook.com/login.php` («Login to Facebook») и уже готов был
доложить «FB разлогинен». **Проверил — это неправда.** FB-лейн ездит на **нативном katana**,
а не на Chrome (`feedback_fb_transport_pixel_app_not_server_proxy`). Нативный FB **жив и
работает**: свип прямо сейчас читает 112 тредов, 36 ждущих, роутит и паркует.
Те Chrome-вкладки — **протухший мусор**, к EV-лейну отношения не имеют. Ложную тревогу снял.

## ПОДТВЕРЖДАЮ БАГ EV-ГРУППЫ (только подтверждение, НЕ чинил)

Да, Босс прав. `ev_group_recurring_post.sh`:

```bash
timeout 1200 python3 projects/fb_social/fb_active.py session --live --text "$TEXT" ...
RC=$?
if [ "$RC" -eq 0 ]; then
    echo "$NOW $IDX" > "$STATE"        # ← слот ротации сожжён
    echo "recurring OK → стейт idx=$IDX"
fi
```

Успех = **только `RC=0`**. Проверки, что пост реально `PUBLISHED`, **нет**. Любой тихий
выход-в-ноль (уступил экран, tab_lock занят, katana не поднялась) = «OK» → `idx` двигается,
`ts` ставится.

**Улика, а не теория:** `.kola_state/ev_recurring_last` = `1784160003 9` — «успех» отмечен
**12.3 часа назад**, при том что группа молчит **151 час**. Слот сожжён без поста.

**И это самоусугубляется:** записанный `ts` запускает анти-дубль-гейт `MIN_GAP_H=8`, поэтому
следующий заход честно скажет «SKIP: last post < 8h» и не попробует. Один фантомный успех
глушит лейн на 8 часов вперёд. В логе это и видно: `[2026-07-13T19:00:01Z] SKIP recurring:
last post 10797s ago (< 8h) — анти-дубль`.
Забавно, что шапка скрипта уже описывает ровно этот баг для ночного окна (и его залатали
частным случаем) — а общий случай остался.
