## 2. ЧЕТЫРЕ КОРНЯ

1. **Путь ответа ОДНОНИТОЧНЫЙ через main.** Лид расписан на worker3, но worker3 **не будит
   НИКТО**: `mail_watch`, `inbox_watcher`, `fb_inbox_sweep`, `node_notif_watch` зовут только
   `_ac.wake(MAIN_AGENT)`; `commit_watchdog` — тоже main. main = единая точка отказа: он висит →
   осиротели лиды ВСЕХ каналов разом.
2. **`stuck` — братская могила.** `commit_watchdog` смотрит только `status=='open'`;
   `agent_capacity.open_lead_commits()` — тоже. После эскалации коммит исчезал из поля зрения
   ВСЕЙ системы, **включая сам ор про сироту**. Молчание читалось как «всё хорошо».
3. **Инфраструктурный сбой ЖРАЛ бюджет лечения.** `tries` рос НЕ СМОТРЯ на возврат `rewake()`:
   `rc=1` (турн не доехал), аварийный стоп и потолок агентов тратили попытку так же, как
   доехавший-но-недоделанный турн. Зубы кусали ГВАРД.
4. **Бюджет меньше аварии.** `MAX_HEAL=2` × 20 мин = 40 мин покрытия против 45-минутного
   зависания main. Не хватало по построению.

## 3. ЧТО СДЕЛАНО

**`lib/lead_guarantee.py` (новый).** Будит **ПОЛОСНОГО ВОРКЕРА НАПРЯМУЮ**, минуя main.
`stuck` снова рабочее состояние (`_revive`). Попытка списывается **только за доставленное
назначение**; гвард → `LAWFUL_REFUSAL` без траты бюджета. Успех доказывается **внешним фактом**
(строка `CARRIED` в watcher.log — её пишет только успешная отправка), а не кодом возврата спавна.
Один лид за прогон: **телефон — один экран**. Всё через `agent_capacity.wake()` (потолок, бронь,
память, стоп, пауза полосы, аудит).

**`scripts/commit_watchdog.py`.** `tries` растёт только при `delivered=True` (корень 3).

**`lib/dirigent_enforce.py`.** Зуб `check_lead_answer_guarantee` — в ТОМ ЖЕ коммите, по стоячему
правилу. Плюс **исполняемая доработка** `rework_fn`: до сих пор FAULT давал лишь СТРОКУ-заметку,
которую никто не исполнял — ровно тот дефект, ради которого модуль и писался.
