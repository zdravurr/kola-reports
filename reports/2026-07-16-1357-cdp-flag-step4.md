# ЧАСТЬ A — CDP

## A1. Моя рекомендация: 24/7 держать CDP НЕ НАДО. И НЕЛЬЗЯ.

Ты спросил мою оценку — вот она, с цифрами, а не «на ощупь».

**CDP физически не может быть поднят постоянно, пока свипы ходят /15 мин.** Замерил
доступность 30 минут (сэмпл /30с): **36 UP / 25 DOWN = 59%**. Механизм пойман в прямом эфире:

```
13:24 тик → cdp_up (проверено)
13:25:07 … 13:31:12  rc=0_UP          ← 13 сэмплов подряд
13:30      свип берёт экран → Chrome в фон
13:31:48   rc=28_DOWN                 ← мост умер ЧЕРЕЗ 1.5 мин после начала свипа
13:31:48 … 13:39:01  DOWN (~7 минут)
13:39 тик → healed_ok → 13:39:31 rc=0_UP
```

Свип **обязан** выводить FB на перёд — это его работа. Chrome при этом уходит в фон, а
Android глушит devtools у фонового Chrome. **Это не баг, который можно «победить»** — это
устройство Android. Keepalive может лишь поднимать мост обратно в паузах. Война /15 мин
невыигрываема по построению.

**И побеждать её не нужно, потому что on-demand УЖЕ РАБОТАЕТ.** Проверил всех трёх
потребителей CDP — каждый поднимает мост сам на старте:

| Потребитель | Когда | Как поднимает |
|---|---|---|
| X-прогрев fin/ca | 10/16/22, 13/20/02 | `x_bridge_ensure.sh` **инлайн в кроне**, перед сессией |
| `cl_renew` | 43 10 */3, 47 10 * * 2 | `x_bridge_ensure.sh` (мой фикс шага 3) |
| `growth_collect` (Огонь) | 0 9 | свой `bridge_up()` — «Same script the warmup uses» |

**Вывод (моя ставка):** гарантия — **on-demand на старте сессии**, она уже есть и работала
всегда (в шаге 1 в логе видно `[bridge] up (try 1)` ровно на старте). Keepalive нужно
считать **дешёвым датчиком доступности**, а НЕ гарантией. Раньше мы принимали его за
гарантию — и потому «healed 40%» читали как успех, хотя это был диагноз.

Поэтому я **не стал вырывать** heal из keepalive (это было бы сверх приказа и лишило бы нас
единственного честного датчика), но и **не стал** раздувать его в 24/7-войну. Минимально:
увёл с минуты коллизии + научил говорить правду.

## A3. Ложный «healed» УБИТ

Было — печатал «healed» **до** проверки:
```bash
if ! curl ... ; then adb connect; adb forward; am start chrome
    echo "[keepalive] ... bridge healed"      # ← НИЧЕГО не проверив
fi
```
Стало — проверяю **фактом** (та же дисциплина, что `_forward_alive()` в `cl_renew`):
```bash
cdp_ok() { curl -s -m4 .../json/version 2>/dev/null | grep -q Chrome; }
if ! cdp_ok; then
    ...; sleep 3
    if cdp_ok; then echo "healed_ok (CDP was down → verified UP)"
    else           echo "heal_FAILED (CDP still DOWN after adb+forward+chrome)"; fi
else               echo "cdp_up (verified, no heal needed)"; fi
```

## ДОКАЗАТЕЛЬСТВО — 4 ПОДРЯД РЕАЛЬНЫХ КРОН-ТИКА


Независимая проверка сразу после 13:54 — **rc=0 три раза из трёх**:
```
probe1 rc=0 "Browser": "Chrome/149.0.7827.200"
probe2 rc=0 "Browser": "Chrome/149.0.7827.200"
probe3 rc=0 "Browser": "Chrome/149.0.7827.200"
```

**Читай честно, а не как победу.** 4/4 тика **заканчиваются** проверенным `rc=0` — это и есть
то, что ты просил. Но **3 из 4** — это `healed_ok`, то есть **мост успевал умереть между
тиками**. Метрика теперь не льстит: она прямо показывает, что 24/7-моста нет и не будет
(59% доступности). Старый код на этих же четырёх тиках напечатал бы «healed» ×3 и ничего бы
не доказал. **Зелёного по «мост стабилен 24/7» не ставлю — такой цели достичь нельзя;
достижимая цель (мост поднят на старте сессии) выполняется on-demand.**
`heal_FAILED` за окно наблюдения — ноль.

## ПОЧЕМУ ФЛАГ НИКОГДА НЕ УМИРАЕТ

```

13:47:45 buyer_waiting ДО свипа: None        (истёк 13:47:17, свип пришёл 13:47:45)
```
**TTL = 900с, период свипа = 900с.** Флаг перевзводится ровно в тот момент, когда должен
умереть. Иногда промахивается на секунды (в 13:47 успел истечь на 28 с) — и тут же ставится
снова. Для 50-минутной X-сессии окно в 28 секунд раз в 15 минут = **флаг вечный**.

*(Мелкая честность: моя же строка «переставляется РАНЬШЕ, чем истечёт» — не всегда верна,
раз в 13:47 он лапнул. Формулировка чуть сильнее факта; сам факт — флаг эффективно вечный.)*
