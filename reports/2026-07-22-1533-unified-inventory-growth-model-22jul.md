━━━ ЧТО ПОСТРОЕНО ━━━
1) lib/inventory_growth.py (НОВЫЙ) — ОДНА живая запись на вещь, сводит то, что раньше жило разрозненно:
   • что/цена/статус/КАТЕГОРИЯ
   • где LIVE + КОГДА ГЛАЗА подтвердили (из listing_coverage, не «claimed»)
   • где ОТСУТСТВУЕТ (missing/stale/draft)
   • last renew/bump на КАЖДОЙ площадке (renewal_ledger)
   • ПРАВИЛА ВИДИМОСТИ площадки — окно затухания: CL 3д / FB 7д / OU 7д / eBay 30д(auto). CL/FB/OU
     награждают свежесть, бамп держит вещь в топе поиска (= обращения)
   • ПРОДВИЖЕНИЕ по КАТЕГОРИИ: какие релевантные группы есть (bike→bike-группы, ev→EV-группа) +
     когда последний раз постили (promotion_ledger)
   triggers(): needs_renew (площадка past decay без бампа) + needs_promotion (у категории есть
   группы, давно/ни разу не постили). ЭТО И ЕСТЬ «велик перестал звонить» — теперь оно ВИДНО и ловится.

2) lib/dirigent_enforce.py — 2 НОВЫХ КОНТРАКТА Дирижёра, driven моделью (проактивно, не по вопросу):
   • listing_aging → rework_fn БАМПАЕТ где renew автоматизирован (CL phone-native, fail-closed по
     замку экрана; двойного бампа нет — cl_renew сам проверяет eligibility). OU/FB — честно surface
     (bump-флоу ещё не построен).
   • promotion_cadence → surface: вещь ждёт перепоста в группы своей категории; аккаунт не член
     bike-групп → сначала вступить, потом постить через fb_active.post_to_group (driver ЗАВЯЗАН).
