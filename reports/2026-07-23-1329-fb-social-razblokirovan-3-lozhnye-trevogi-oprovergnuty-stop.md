## P2 — fb_social РАЗБЛОКИРОВАН. НО ПРИЧИНА НУЛЯ ПОСТОВ БЫЛА ДРУГАЯ

### 2а. Канал доставки фикса был закрыт ТРЕМЯ слоями — снял все три

Твоё предложение отработало ровно так, как ты сказал: **фикс не пришлось копировать — я
перенёс сам путь ВНУТРЬ песочницы.**

1. **ПУТЬ.** workspace физически на томе `/mnt/volume.../home/botuser/.openclaw`, а в домашнем
   каталоге стоял **симлинк**. Песочница агента (`claude -p`, cwd=/home/botuser) резолвит
   симлинк → путь уезжает в `/mnt`, вне разрешённого дерева. Разбор говорил правду.
   → Симлинк заменён на **systemd BIND-монт** `home-botuser-.openclaw.mount`.
   **Копию не делал сознательно:** двум живым копиям кода разойтись нечем только когда копия
   ОДНА. Здесь один инод доступен по двум путям — старый `/mnt`-путь жив для всей инфры
   (ничего не переезжало, ничего не ломалось), а `realpath` теперь внутри `/home/botuser`.
   Данные остались на `/dev/sda` (на корне всего 14G — поэтому симлинк и был).
   Fail-safe: не смонтирован том → точка монтирования root:root 0555, botuser не запишет
   ничего «мимо» — громкий отказ вместо тихой записи стейта не туда.
2. **ПРАВА.** **56 файлов workspace принадлежали root** — мои же правки из root-сессий, среди
   них `projects/fb_social/fb_active.py`, РОВНО тот файл, куда 8 суток не мог лечь фикс.
   Даже с починенным путём канал остался бы закрыт по правам, и разбор снова честно сказал бы
   «невносим». → Владение возвращено botuser + сторож `workspace_owner_guard.sh` (крон root,
   каждые 10 мин): улика механическая, а не «я не забуду chown». Проверен живьём.
3. **РЕЖИМ.** `claude -p` без `--permission-mode` авто-отклоняет Edit/Write вообще везде.
   → Доказано живьём: подагент от botuser с `acceptEdits` из cwd=/home/botuser записал файл
   в workspace (проба `SANDBOX_OK_2026-07-23`).

### 2б. А НУЛЬ ПОСТОВ ДЕРЖАЛ МОЛЧАЛИВЫЙ ПУТЬ — В ЛОГЕ ОШИБОК ЕГО НЕТ ВООБЩЕ

Девять разборов называли причиной `subprocess.TimeoutExpired` — «единственный класс ошибок в
логе». Он реален, но **простой держал не он.** Механика, целиком из логов:

```
post_watchdog видит «FB-группа просрочена» → перезапускает ev_group_recurring_post.sh
   → навигация в группу падает (post_nav_fail)
   → fb_active выходит с кодом 0  ← у active_session НЕТ ни return, ни sys.exit
   → обёртка пишет в стейт «пост был только что»
   → анти-дубль (MIN_GAP_H=8) глушит ВСЕ заходы 8 часов
   → в том числе ПЛАНОВЫЙ пост 22.07 19:00: «SKIP recurring: last post 25200s ago»
   → сторож перезапускает снова → цикл замкнут сам на себя
```

Улики: последний `post_PUBLISHED` — **20.07 00:05** (85 ч). После него только
`post_nav_fail`: 21.07 12:02, 22.07 11:55, 22.07 12:03, 22.07 20:02, 23.07 12:01.
А `ev_recurring_last` показывал **23.07 12:00** — то есть стейт был обновлён провалившимся
прогоном ровно за 30 минут до того, как я его открыл.

**Что починено в коде:**
- **Контракт кода выхода:** `0` = пост ПОДТВЕРЖДЁН, `3` = пропуск (гейт / клиент ждёт / телефон
  спит / кап — стейт не трогать, прийти позже), `1` = провал этапа. Обёртка двигает стейт и
  ротацию блока ТОЛЬКО на 0.
- **Анти-дубль судит по факту публикации** — `post_PUBLISHED` в `fb_active_log.jsonl`, а не по
  нашей записи о собственной попытке. Проверено: отдаёт 20.07 00:05, разрыв 85ч → не глушит.
- **Навигация:** `open_group()` теперь идёт ПЕРВЫМ детерминированным путём `open_joined_group`
  («Your groups → See all», доказан живьём 22.07 на bike-группе) и требует КОМПОЗЕР на экране;
  флаки-поиск FB остался фолбэком. Именно поиск давал все `post_nav_fail`: порядок выдачи
  пляшет, результаты litho-рендерятся и в uidump их нет.
- **SSH-таймаут (тот самый) — тоже закрыт:** у `su -c` не было УДАЛЁННОГО timeout'а, поэтому
  зомби висел НА ТЕЛЕФОНЕ и ловил следующий вызов; а `TimeoutExpired` летел наружу и ронял
  заход целиком — отсюда «пустой 24ч-лог = ложный green». Теперь `timeout N su -c`,
  BatchMode + ServerAliveInterval, один ретрай, `DeviceUnreachable` → мягкий пропуск, и
  `device_healthy()` ДО захвата экрана (проверен живьём).
