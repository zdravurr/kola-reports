# Pixel SSH починен + АРХИТЕКТУРА: ты прав, туннель — не тот путь — 2026-07-15

Честный разбор по твоим вопросам + что реально сделано и что дальше.

## ЧТО ПОЧИНЕНО (proven)
1. ✅ **Pixel sshd вычищен**: было 11 процессов (1 мастер + 10 зависших sshd-session от моих
   `ssh -D` — они и забили слоты). Убил всё через ADB, перезапустил свежий sshd. `:8022`
   слушает. Сейчас 9 сессий — но с запасом (см. п.2), не заклинено.
2. ✅ **sshd закалён**: `MaxStartups 60:30:200`, `MaxSessions 40` (было по умолчанию 10 — из-за
   этого полу-открытые туннели и морили sshd). Автозапуск на буте уже есть (Termux:Boot
   `pgrep sshd || sshd`). Это чинит НЕ только CL — весь телефонный транспорт (тёплый mux/прогрев)
   теперь здоровее.
3. ✅ Гигиена хоста: убил свои зависшие `ssh -D`, снёс протухший mux-сокет.

## ТВОЙ ВОПРОС ПРО АРХИТЕКТУРУ — ТЫ ПРАВ (эмпирически подтверждено)
- **5G-IP телефона — чистый резидентный IP, CL его НЕ банит.** Телефон сейчас на 5G, дом-Cox-WiFi,
  под который строили прокси, ВООБЩЕ не при делах.
- **Туннель server→SSH→Pixel→SOCKS — переусложнён И это флаки-корень.** После починки sshd
  туннель ПОДНЯЛСЯ (ssh -D exit=0, :1080 слушает) — НО трафик сквозь него НЕ идёт: curl через
  SOCKS упал по таймауту (exit=97, IP не вернулся). То есть даже «починенный» туннель не
  доставляет трафик через телефон. Подтверждает твою правоту: этот путь ненадёжен.
- **Правильная архитектура — НАТИВНО НА ТЕЛЕФОНЕ:** гнать публикацию через СОБСТВЕННЫЙ Chrome
  телефона по CDP/ADB (ровно тот путь, которым X-прогрев работает КАЖДЫЙ день) — это и есть
  выход через мобильный IP телефона, БЕЗ SSH-туннеля. adb-forward к Chrome (:19222) — на ADB,
  который жив и стабилен, а не на флаки SSH.
